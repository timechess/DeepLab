# 工作报告

该工作流根据触发时用户行为状态与快照的增量，形成用户行为报告，发送给LLM生成工作报告。

用户行为主要分以下几类：

1. 任务的创建/完成
2. 精读报告评论的创建/编辑
3. 笔记的创建/编辑

快照存储了上次工作流运行时的行为数据，再次运行时需对比当前数据库中行为数据与快照的差异，具体需要收集如下内容：

1. 新创建的任务
2. 新完成的任务
3. 新增的评论内容
4. 修改的评论的改动
5. 新创建笔记的全部内容
6. 编辑已有笔记的改动

将以上内容格式化为可读的Markdown后插入到发送给LLM的提示词中，默认提示词如下：

系统提示词：
```markdown
你是严谨的科研工作日报 Agent。你将基于用户前一天的真实行为记录生成中文 Markdown 日报。
禁止虚构输入中不存在的信息，内容要具体、可执行。
```

用户提示词：
```markdown
请基于以下“用户行为汇总”生成 {{BUSINESS_DATE}} 的工作日报。

你必须严格输出且只输出以下三节（Markdown 标题）：
## 昨日工作总结
要求：针对输入中的真实行为给出精简分条列点（`-` 列表），强调已完成事项、关键进展、阻塞点。

## 今日工作规划
要求：结合“当前未完成任务 + 昨日进展”给出今日计划，分条列点，明确优先级和可执行动作。

## 工作建议
要求：给出面向当前研究与工程推进的建议，可包含研究创新点、验证路径、实验设计、风险控制等，同样分条列点。

约束：
1. 所有结论必须能在输入中找到依据，不得杜撰。
2. 不要输出除上述三节外的其他大标题。
3. 输出必须是 Markdown 正文，不要包裹代码围栏。

【业务日期】
{{BUSINESS_DATE}}

【来源日期标记】
{{SOURCE_DATE}}

【用户行为汇总】
{{ACTIVITY_MARKDOWN}}
```

该工作流也需要在首页占据一个模块，放在论文推荐下面，显示当前行为相比于快照的增量，无增量则无法触发工作流，当天已有报告也无法触发新的工作流。展示日报需进行markdown渲染。新增一个`/work_report`页面用于管理已经生产的报告，类似精读报告的处理，也需要有详情页。
